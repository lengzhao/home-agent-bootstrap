# 技能维护

维护家庭助手技能时，把目标放在“让未来的 Agent 更稳定地做对事”。技能应小、清晰、按需触发，不要把长期记忆、密钥或大段项目文档塞进技能。

## 开始前

- 先阅读 `skills/` 下相关 `SKILL.md`，判断已有技能是否能覆盖需求。
- 如果涉及 cc-connect、微信、daemon、cron 或文件回传，优先检查 `skills/cc-connect/SKILL.md` 是否应更新，而不是新建重复技能。
- 明确用户希望沉淀的是触发条件、执行步骤、安全检查、常用命令，还是失败处理。
- 修改已有技能时保留原 `name`，除非用户明确要求重命名或合并。

## 何时新建技能

- 某类请求会重复出现。
- 某个流程有固定步骤或安全检查。
- 某个外部工具需要固定调用方式。
- 用户明确要求“以后记住这个流程”或“做成技能”。
- 现有技能无法自然覆盖，强行加入会让原技能过宽。

## 何时合并或重写

- 两个技能触发场景高度重叠。
- 一个技能只是另一个技能的子步骤。
- 用户会难以判断该用哪个技能。
- heartbeat 或日常使用发现多个技能给出冲突指令。
- 技能正文堆积了过时命令、重复说明或缺少安全边界。

合并后说明保留哪个技能、移除了什么重复内容，以及未来应该触发哪个技能。不要为了兼容未发布的旧草稿保留多套相同流程。

## 存放位置和命名

新技能默认放在：

```text
skills/<skill-name>/SKILL.md
```

技能名使用小写字母、数字和连字符，和目录名保持一致。不要创建额外 README、CHANGELOG 或安装说明，除非用户明确需要给人阅读的交付文档。

## 技能结构

每个技能必须包含简洁 frontmatter，只保留 `name` 和 `description`：

```markdown
---
name: example-skill
description: 说明这个技能做什么，以及用户出现哪些请求时必须使用。
---
```

`description` 是触发入口，要写清楚具体场景、近义表达和边界。正文只在触发后才会加载，所以不要把关键触发条件只写在正文里。

正文优先包含：

- 适用场景。
- 必要前置条件。
- 执行步骤。
- 安全边界。
- 常用命令或工具。
- 失败处理或需要询问用户的情况。

## 编辑原则

- 不要把所有内容堆到一个大技能里。
- 不要重复已有技能。
- 不要把简单常识写成冗长说明，只保留未来 Agent 不容易自动知道的流程、约束和命令。
- 对易错、脆弱或高风险流程给更明确的步骤；对写作、整理、问答类流程保留判断空间。
- 涉及高风险家庭动作的技能必须写明二次确认规则。
- 技能不能保存密钥、token、账号密码。
- 技能不能保存家庭成员敏感隐私；长期偏好和家庭资料应放在工作目录的 Markdown 或受控本地数据文件中。

## 完成检查

- `name` 与目录名一致。
- `description` 足够具体，能帮助未来 Agent 决定是否触发。
- 正文短而可执行，没有和其他技能重复的大段内容。
- 高风险动作、隐私数据、外部发送、删除或重启都有确认规则。
- 修改或合并后，向用户说明变更原因和使用上的变化。
